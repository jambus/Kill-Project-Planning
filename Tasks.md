# 智能排期优化任务清单

## 高优先级任务（影响用户体验）

### Task 1: 优化AI调用效率 - 批量/单次规划
**优先级**: P0  
**预估工作量**: 3-5天  
**当前问题**:
- 每个项目单独调用一次AI API，10个项目需要10次网络往返
- 累积延迟导致用户等待时间过长
- API成本线性增长

**优化方案**:
- **方案A**: 单次AI调用规划所有项目（推荐）
  - 修改prompt，让AI一次性返回所有项目的分配方案
  - 优点：最快、成本最低
  - 缺点：prompt更长，需要更强的AI推理能力
  
- **方案B**: 批量处理（3-5个项目一批）
  - 折中方案，平衡prompt长度和调用次数
  - 优点：降低单次prompt复杂度
  - 缺点：仍需多次调用

**预期收益**:
- 排期速度提升 80-90%
- API成本降低 80-90%
- 用户体验显著改善

**技术要点**:
- 修改 `scheduleWithAI` 函数结构
- 重新设计AI响应格式（返回多项目分配）
- 调整JSON解析逻辑

---

### Task 2: 增强错误处理和重试机制
**优先级**: P0  
**预估工作量**: 1-2天  
**当前问题**:
- AI响应解析失败时直接跳过项目
- 没有重试机制
- 用户不知道哪些项目排期失败

**优化方案**:
1. 添加重试逻辑（最多3次，指数退避）
2. 记录失败的项目和原因
3. 在UI中显示排期结果摘要（成功/失败/部分完成）
4. 提供fallback策略（简单轮询分配）

**预期收益**:
- 提高排期成功率
- 增强系统鲁棒性
- 改善用户反馈

**技术要点**:
```typescript
interface ScheduleResult {
  success: boolean;
  completedProjects: string[];
  failedProjects: Array<{id: string, reason: string}>;
  partialProjects: Array<{id: string, allocated: number, required: number}>;
}
```

### Task 3: 添加并发控制和状态管理
**优先级**: P1  
**预估工作量**: 0.5-1天

- 没有loading状态锁

**优化方案**:
1. 添加全局排期状态标志
2. 排期进行时禁用触发按钮
3. 显示进度指示器（当前处理第X/Y个项目）
4. 支持取消正在进行的排期

**预期收益**:
- 防止重复操作
- 更好的用户反馈
- 避免资源浪费

## 中优先级任务（降低成本和优化性能）

### Task 4: 优化Prompt结构和Token使用
**优先级**: P1  
**预估工作量**: 2-3天  
**当前问题**:
- 每次调用都发送完整的资源列表和规则说明
- 大量重复信息导致token浪费
- Prompt结构不够清晰

**优化方案**:
1. 使用结构化messages（system + user）
2. 将静态规则移到system message
3. 只在user message中发送变化的数据
4. 如果API支持，启用prompt caching
5. 精简资源描述，只发送必要字段

**示例结构**:
```typescript
{
  system: `你是资源调度专家。规则：
    1. 工作日计算：周一到周五，排除法定节假日
    2. 角色限制：前端/后端/APP/全栈只做dev，测试工程师只做test
    3. 不能超分配资源
    ...`,
  user: `当前状态：
    资源：[{name: "张三", role: "前端", idleDays: 5}, ...]
    项目X：需要前端3天，后端5天
    请分配资源并返回JSON`
}
```

**预期收益**:
- Token使用量降低 40-60%
- API成本显著下降
- Prompt更易维护

### Task 5: 改进AI约束传递机制
**优先级**: P1  
**预估工作量**: 1-2天  
**当前问题**:
- AI不知道分配会被 `Math.min()` 截断
- AI建议经常被忽略，浪费推理能力
- 无法充分利用AI的决策能力

**优化方案**:
1. 在prompt中明确告知每个资源的**当前可用天数**
2. 告知项目的**剩余需求天数**
3. 让AI在约束内做决策，而不是事后截断
4. 添加验证：如果AI建议超出约束，记录warning

**Prompt改进示例**:
```
项目A：还需前端开发3天
可用资源：
- 张三（前端）：当前空闲5天，可分配最多5天
- 李四（前端）：当前空闲2天，可分配最多2天
请在约束内分配，确保不超过可用天数
```

**预期收益**:
- AI建议更准确
- 减少无效推理
- 提高分配质量

---

### Task 6: 性能优化 - 资源状态缓存
**优先级**: P2  
**预估工作量**: 1-2天  
**当前问题**:
- 每次分配后重新计算所有资源的空闲时间
- O(n*m) 复杂度，大量重复计算

**优化方案**:
1. 维护资源状态Map，存储每个资源的当前空闲时间
2. 分配后只更新相关资源的状态
3. 使用增量更新代替全量重算

**技术要点**:
```typescript
interface ResourceState {
  id: string;
  idleDays: number;
  nextAvailableDate: Date;
  currentAllocations: Allocation[];
}

const resourceStateCache = new Map<string, ResourceState>();
```

**预期收益**:
- 计算性能提升 50-70%
- 大项目排期更快

interface HolidayConfig {

**预期收益**:

### Task 8: 添加验证和审计功能
**优先级**: P2  
**预估工作量**: 2-3天  
**当前问题**:
- 排期完成后没有验证逻辑
- 无法确认分配是否正确
- 缺少排期报告

**优化方案**:
1. 后置验证函数
   - 检查总分配工作量 vs 项目需求
   - 验证资源是否过载
   - 检测日期冲突
2. 生成排期摘要报告
   - 成功完成的项目
   - 部分完成的项目（原因）
   - 失败的项目（原因）
3. 资源利用率统计
   - 每个资源的负载率
   - 空闲资源识别
   - 瓶颈资源识别

**UI展示**:
```
排期结果：
✅ 完成：8个项目
⚠️ 部分完成：2个项目（资源不足）
❌ 失败：0个项目

资源利用率：
- 张三（前端）：95% 🔴 高负载
- 李四（后端）：60% 🟢 正常
- 王五（测试）：30% 🟡 低负载
```

**预期收益**:
- 提高排期可信度
- 帮助识别资源瓶颈
- 改善决策支持

### Task 10: AI响应格式标准化
**优先级**: P3  
**预估工作量**: 1天  
**当前问题**:
- 使用正则表达式解析JSON，比较脆弱
- 没有schema验证
- AI可能返回不符合预期的格式

**优化方案**:
1. 如果API支持，使用JSON schema约束输出
2. 添加响应格式验证（使用zod或类似库）
3. 更robust的JSON提取逻辑
4. 在prompt中提供明确的输出示例

**技术要点**:
```typescript
import { z } from 'zod';

const AllocationSchema = z.object({
  resourceId: z.string(),
  resourceName: z.string(),
  days: z.number().min(0),
  role: z.enum(['前端', '后端', 'APP', '全栈', '测试工程师']),
  reasoning: z.string().optional()
});

const AIResponseSchema = z.object({
  allocations: z.array(AllocationSchema)
});
```

**预期收益**:
- 更可靠的解析
- 更好的错误提示
- 类型安全

## 风险和注意事项

1. **API兼容性**: 确认使用的AI API是否支持prompt caching、JSON schema等特性
2. **向后兼容**: 优化时保持数据结构兼容，避免破坏现有数据
3. **测试覆盖**: 每个优化都需要充分测试，特别是Task 1（核心逻辑变更）
4. **渐进式优化**: 不要一次性改动太多，每个task独立验证
5. **用户反馈**: 优化后收集用户反馈，验证改进效果
